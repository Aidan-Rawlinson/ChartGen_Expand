## Status: Second database (Indicators toolkit) built end-to-end through data acquisition; credentials relocated to Config tab; chart rendering for the new shape deliberately deferred to next session

### What works (built this session)

- **Credentials relocated to Config tab.** The app-level login gate is gone entirely — no authentication required to launch, create, open, or save a workfile. A single credentials box ("NHS Annual and Indicator Toolkits": email, password, Validate button) lives in the Config tab instead, validated on demand. Confirmed mid-session: one shared token authorises both the NHS and Indicators APIs — no per-database credential sets needed. Fetch fails soft (clear message, no crash) if no valid token exists. Save attribution (`last_saved_by`) is blank rather than defaulted when no credentials have been validated this session.
- **`TimeSeries` — the fourth canonical data shape** (`core/shared/normalisation_containers/shapes/timeseries.py`). One or more Metric-Series, one value per unit per metric *per period*, across a period axis shared once by the whole shape (not per metric — a deliberate correction mid-session: a data shape is one dataset, and if periods didn't hold across every metric in it, it wouldn't be one shape). Stats recomputed locally per period, same convention as every other shape; the API's own supplied stats (mean/median/national-average) are dropped entirely, never stored. Deliberately NOT yet wired into `dispatch.py`/`build_population_layers` — no chart type references it yet, so nothing calls those generic dispatch points with a TimeSeries instance. `cache_reader.py` deserialises it correctly, so the Charts tab identifies it and stops cleanly ("No Base Charts defined for shape type 'TimeSeries'") rather than crashing.
- **`core/acquisition/toolkit_indicators/`** — new package mirroring `toolkit_nhs/`'s shape: `api_client.py` (separate API host, `icsapi.nhsbenchmarking.nhs.uk`), `url_parser.py` (real-URL-confirmed tier-id cascade `o`→`d`→`c`→`b`, mirroring the source VBA exactly, including its graceful "no `date=`" fallback), `table_naming.py` (`submissions_timeseries_{project_id}`, no year component), `population_tables.py`, `transformers.py`, `fetch.py`.
- **`core/acquisition/url_triage.py`** — new; classifies a chart URL as `"nhs"`/`"indicators"` by path shape alone (`/outputs/{id}` vs `/project/{id}/toolkit`, confirmed against real URLs — same front-end domain, path is the only reliable signal). Wired into both manifest-row creation routes (template extraction and direct Excel entry), so yellow-box and direct-entry URLs get identical triage.
- **`core/acquisition/fetch_dispatch.py`** — new; the Imports tab's Fetch button now calls this instead of `toolkit_nhs.fetch` directly. Routes each manifest row to the correct toolkit by its `database` column and reports progress as one continuous total across both — every URL treated identically regardless of database (fixed mid-session; originally shipped as two progress phases, corrected on request).
- **Second population-table maintenance model, deliberately different from the NHS one.** `submissions_timeseries_{project_id}` merges on *every* fetch, not built once and skipped thereafter — a single Indicators report response already spans a project's full period history, and submissions genuinely drop in and out of the population over time (confirmed). `Region()` is carried on these rows too (keeping the identical-headers convention every population table follows), sourced the same way the NHS side sources it — a lookup against the organisation row, not chart data — just looked up against `nhs_organisations` directly at merge time, since the Indicators API supplies no region field of its own.
- **Organisation-identity assumption, confirmed for now.** Indicators submissions link to `nhs_organisations:{organisation_id}` on the assumption that organisation ids match across both APIs — explicitly agreed as an assumption to revisit if wrong, not a verified fact.
- **`cache_writer.py` relocated** from `toolkit_nhs/` to `shared/infrastructure/` — audited as having no NHS-specific logic at all, so both toolkit packages import the same module rather than duplicating it.
- **All governed docs current** — Architecture (new Decision 10, package tree, prose rows), Functional Spec (new §7.4, "four" not "three" shapes, TimeSeries in the shapes table), Feature List (credentials rows corrected, Indicators population-table rows added, "TimeSeries chart rendering — Not built" added), Glossary (Package Map, new Indicators toolkit and Period entries) all updated to match. Primer untouched (edit-locked).

- **Correction to an earlier session claim — `format_modifier` retrofit is narrower than stated mid-session.** Checked against actual code before writing this up (Maintenance Guide ground truth discipline): `format_modifier` is already populated correctly from the API for NumericSeries and NumericCompositional (`transform_bar_chart`/`transform_radar_chart` in `toolkit_nhs/transformers.py` both set it from `formatModifier`). Only `CategoricalCompositional` lacks it entirely — no field on the dataclass, not picked up by `transform_pie_chart`/`transform_yn_chart`. The outstanding work is a `CategoricalCompositional`-only gap, not a three-shape retrofit as previously described to the user.
